# SlateDrop — Platform Architecture

**Created:** 2026-05-02  
**Status:** LOCKED — This is the authoritative SlateDrop architecture reference.  
**Replaces:** Any prior descriptions of SlateDrop as "a file upload page" or "storage tab."

---

## 1. What SlateDrop Is

SlateDrop is the **central file system, storage routing, deliverable history, and sharing infrastructure** for the entire Slate360 ecosystem.

It is NOT:
- A simple file upload area
- A tab that shows uploaded images
- Isolated cloud storage for one app

It IS:
- The Finder / File Explorer / Dropbox-level storage layer that all Slate360 apps write to
- The canonical location for every deliverable, report, proposal, and polished output
- The intake system for received files and collaborator submissions
- The share-link origin for external viewers (clients, stakeholders, public)
- An entitlement-aware folder hierarchy that evolves as subscriptions change

Every subscribed user has SlateDrop. Every app writes its deliverables to SlateDrop. SlateDrop is the connective tissue of the platform.

## 2. SlateDrop Folder Hierarchy

### Root Structure (per organization)

```
SlateDrop (org root)
├── Field Projects/
│   └── [Field Project Name]/
│       ├── Site Walk/
│       │   ├── In Progress/
│       │   ├── Photos/
│       │   ├── Plans/
│       │   ├── Reports/
│       │   └── Deliverables/
│       └── Shared/
│
├── Projects/
│   └── [Project Name]/
│       ├── Site Walk/
│       │   ├── In Progress/
│       │   ├── Photos/
│       │   ├── Plans/
│       │   ├── Reports/
│       │   └── Deliverables/
│       ├── 360 Tours/
│       ├── Design Studio/
│       ├── Content Studio/
│       ├── RFIs/
│       ├── Submittals/
│       ├── Budgets/
│       ├── Schedules/
│       ├── Correspondence/
│       ├── Closeout/
│       └── Shared/
│
├── App Folders/
│   ├── Site Walk/          (standalone deliverables / shared items not tied to a project)
│   ├── 360 Tours/          (standalone tours / public hosted links)
│   ├── Design Studio/      (standalone design files)
│   └── Content Studio/     (standalone media clips)
│
├── Shared With Me/
├── Received/               (external uploads, intake form submissions)
├── Sent/                   (deliverables and links sent to contacts/clients)
└── Archive/                (expired or retired project folders — data preserved)
```

### Folder Generation Rules

Folders are auto-generated by the system based on entitlements and project type. Users should NOT manually create the root hierarchy. The system creates folders:

| Trigger | Folders Created |
|---|---|
| User subscribes to Site Walk | `App Folders/Site Walk/` + Shared With Me / Received / Sent / Archive |
| User creates a Field Project | `Field Projects/[Name]/Site Walk/` tree |
| User creates a full Project | `Projects/[Name]/` full tree including all app sub-folders |
| User adds 360 Tours subscription | `App Folders/360 Tours/` + `Projects/[Name]/360 Tours/` for each existing project |
| User adds Design Studio | `App Folders/Design Studio/` + project sub-folders |
| Subscription changes / cancels | Folders remain visible (read-only); data is preserved; creation of new items in-app is blocked by entitlement |

**Data preservation rule:** Canceling a subscription must NEVER delete or hide prior data. The folder tree becomes read-only. The user can still view, download, and share existing files. New items cannot be added until the subscription is renewed.

## 3. File Routing by Operation

Each operation in any app must write to a predictable SlateDrop path.

### Site Walk — File Routing

| Operation | SlateDrop Path |
|---|---|
| Photo captured in a walk | `Field Projects/{id}/Site Walk/Photos/` or `Projects/{id}/Site Walk/Photos/` |
| Walk marked In Progress | Record in `Field Projects/{id}/Site Walk/In Progress/` |
| Report generated | `Field Projects/{id}/Site Walk/Deliverables/{deliverable-id}/` |
| Proposal generated | `Field Projects/{id}/Site Walk/Deliverables/{deliverable-id}/` |
| Plans uploaded | `Field Projects/{id}/Site Walk/Plans/` |
| Standalone deliverable (no project) | `App Folders/Site Walk/{deliverable-id}/` |

### Cross-App — File Routing

| Operation | SlateDrop Path |
|---|---|
| 360 Tour published | `Projects/{id}/360 Tours/{tour-id}/` or `App Folders/360 Tours/{tour-id}/` |
| Design file exported | `Projects/{id}/Design Studio/{export-id}/` |
| Content clip rendered | `Projects/{id}/Content Studio/{clip-id}/` |
| Client-facing share link | `Sent/` record referencing the source deliverable path |
| External intake received | `Received/{upload-id}/` |

## 4. Database Model

SlateDrop folder records live in `project_folders` (not `file_folders` — that table is deprecated).

### Key Tables

| Table | Purpose |
|---|---|
| `project_folders` | Folder nodes in the SlateDrop hierarchy |
| `slatedrop_uploads` | File/object records with metadata, path, size, mime type |
| `slate_drop_links` | Share link records (token, expiry, access count, recipient) |

### Storage Backend

All files are written to Cloudflare R2 (primary) or AWS S3 (fallback) via `lib/s3.ts`.  
The S3-compatible client abstracts both backends. Never write directly to R2/S3 — always go through `lib/s3.ts`.

Presigned upload URLs are issued by `/api/slatedrop/upload` routes which:
1. Verify auth with `withAuth()`
2. Run metering guard before issuing the URL
3. Create the `slatedrop_uploads` record with status `pending`
4. Update status to `complete` after the client confirms upload

## 5. Entitlement-Aware Behavior

SlateDrop adapts based on what the user is subscribed to.

| Entitlement | SlateDrop Behavior |
|---|---|
| No subscription | Cannot create new items; can view received items; can access shared-with-me |
| Site Walk standard | Field Projects tree; App Folders/Site Walk; Received; Sent; Archive |
| Site Walk + 360 Tours | Adds 360 Tours sub-folders in all projects |
| Business / Enterprise | Full project tree including PM folders (RFIs, Submittals, Budgets, etc.) |
| Collaborator (no subscription) | Can view files shared with them; can upload to Received only |

The entitlement check happens server-side at API routes. The UI reflects entitlements by showing/hiding sections. No client-side entitlement bypass is acceptable.

## 6. Sharing and External Links

SlateDrop is the origin of all external share links in Slate360.

Share link lifecycle:
1. User creates a share link from a deliverable or folder
2. System generates a token and creates a `slate_drop_links` record
3. Token is embedded in a URL: `slate360.ai/share/{token}` or `slate360.ai/view/{token}`
4. Recipient clicks the link → no login required → view/download gated by link settings
5. Link can be set to: one-time view, time-limited, max-view count, password-protected (future)
6. Creator can revoke the link at any time
7. Access events are logged for audit trail

## 7. What SlateDrop Must NOT Do

- Must not force users to manually build the root folder hierarchy
- Must not delete data when subscriptions change (only restrict creation)
- Must not allow direct R2/S3 access without going through the API layer and metering guard
- Must not expose other organizations' folders (strict org_id RLS on all tables)
- Must not be confused with the capture-side offline queue (IndexedDB); SlateDrop is the cloud destination, not the local queue

## 8. Integration Points

Every app that saves a deliverable MUST write to SlateDrop using the routing rules in §3. This is enforced by convention and code review, not by a technical lock. Any API route that saves a file and does NOT write to SlateDrop is incomplete.

**Site Walk integration:** Site Walk deliverable generation routes must:
1. Create the deliverable in `site_walk_deliverables`
2. Generate the PDF/interactive output
3. Write the output to SlateDrop via the file routing rules
4. Create a `slate_drop_links` record if the deliverable has a share link
5. Update the deliverable record with the SlateDrop path and share token
