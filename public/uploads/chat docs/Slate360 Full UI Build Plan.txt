Part 2: Design System Architecture

Two-Layer Token System (the unification)

The Slate360 design system has two legal contexts that share a foundation but diverge in accent:

Layer 0 — Shared Foundation (both surfaces)

Canvas:       #0B0F15
Surface:      #0D1219
Alt Surface:  #101722
Heading:      #FFFFFF
Body Text:    #F8FAFC
Muted Text:   #A3AED0
Border Soft:  rgba(255,255,255,0.08)
Border Bright: rgba(255,255,255,0.14)
Font (UI/body): Inter
Font (heads/labels): Space Grotesk

Layer 1a — Marketing tokens (public pages: /, /site-walk, /twin-360, /pricing)

Primary accent:    #00E699 (Green — Site Walk, action, field capture)
Secondary accent:  #3D8EFF (Blue — Twin 360, data, models)
Green Glow:        rgba(0,230,153,0.08)
Blue Glow:         rgba(61,142,255,0.08)
Feature dot: green for Site Walk tiles, blue for Twin 360 tiles

Layer 1b — Authenticated tokens (app shells, dashboard, product surfaces)

Primary accent:    amber-500 (#F59E0B) — CTA, active nav, interactive
Active state:      amber-500/10 bg + text-amber-500
Status success:    emerald-400
Status warning:    amber-400
Status error:      red-400

Resolution rule: The homepage and all public marketing pages use Layer 1a (green/blue). The authenticated app (dashboard, /app, /site-walk, /twin-360 product surfaces) uses Layer 1b (amber). Both sit on the same #0B0F15 canvas and share typography.

The new components/marketing-v2/marketing-v2-tokens.ts will be the authoritative Layer 1a token file. The existing components/mobile-system/mobileTokens.ts is the Layer 1b authority.

Part 5: Complete Build Rules & Safety Mechanisms

This section is the authoritative rule set for any AI working on this repo. Every rule below is enforced — either by automated guard scripts, git hooks, or explicit owner policy. Violations cause build failure, type errors, or roll-back requests.



Rule 1 — File Line Limit (HARD CEILING)

Every file inside app/, components/, or lib/ must stay strictly under 300 lines. Target is ≤150 lines.





At 250 lines, extract logic into dedicated leaf components, custom hooks, or type files.



No exceptions. Files over 300 lines block the architecture guard: npm run guard:architecture.



Orchestrator files (root homepage component, shell wrappers) should be 60–80 lines. They compose — they do not implement.



Rule 2 — Token-Only Colors (NO HARDCODED HEX IN COMPONENTS)

All color values must come from token files. Never write a hex value, rgba(), or Tailwind palette class directly in a component or page file.

Token files:





Marketing/public: components/marketing-v2/marketing-v2-tokens.ts (Layer 1a)



Authenticated/product: components/mobile-system/mobileTokens.ts (Layer 1b)



Global CSS variables: app/globals.css → --s360-* tokens

Banned in component files:





#00E699, #3D8EFF, #0B0F15, any hex value



bg-black, bg-slate-900, bg-zinc-*, text-orange-*, text-amber-* (naked palette classes)



rgba() inline style values



style={{ color: ... }} or style={{ background: ... }}

Correct pattern:

// Wrong
<div className="bg-[#0B0F15] text-[#00E699]">
// Correct — import from token file
import { V2_PAGE_ROOT, V2_GREEN_ACCENT } from "@/components/marketing-v2/marketing-v2-tokens";
<div className={V2_PAGE_ROOT}>



Rule 3 — No Vibe-Coded Content (ZERO TOLERANCE)

These strings are permanently banned from any file inside app/, components/, or any user-facing surface:





Coming Soon



beta / Beta



demo / Demo



placeholder / Placeholder



test / Test (except in e2e/ and *.test.ts files)



Lorem ipsum



fake / mock / filler



Non-actionable metrics (a metric displayed that the user cannot click or act on)



Dead buttons (buttons that have no handler or route to a 404)

V1 is a Foundational Release — not a beta. The app reviewer from Apple or Google must see a complete, working product.



Rule 4 — Design System Identity Rules

Marketing/public surfaces (/, /site-walk, /twin-360, auth pages):

Secondary accent: #3D8EFF (blue — Twin 360, data)

Rule 5 — Mobile Layout Rules (MANDATORY)

Banned on mobile content blocks:





h-[100dvh] and h-screen on content-heavy containers



snap-start on any block that has real scrollable content



fixed bottom nav (use in-flow shrink-0 instead)



h-full inside an overflow-y-auto parent (WebKit resolves as auto, breaks 1fr grid rows)

Required patterns:





Shell: flex flex-col overflow-hidden on <main>



Panels: flex-1 min-h-0 so flex propagates height



Panel scroll: overflow-y-auto on the panel, not the page



Bottom nav clearance: in-flow shrink-0 nav; flex-1 main content fills remaining space



Touch targets: minimum 44px height on all interactive controls



Horizontal padding: px-4 for content; px-6 for sections

Mobile vs Desktop fork rule:





Desktop (≥1024px): asymmetric grid, 55% text / 45% media, lg:h-[100dvh] lg:snap-start



Mobile (<1024px): fluid scroll, w-full h-auto py-16 px-6 block

The question of whether it can build everything is really a question of context window and session continuity — the scope is large enough that it will require multiple focused sessions per phase rather than one single prompt.

What the spec covers for each of the three shells:

Slate 360 Platform Shell (/app)

Twin 360 App Shell (/digital-twin)

Identical structure to Site Walk — header, 2×2 grid, expandable dock — just different content
Accent shifts from #00E699 green → #3D8EFF blue everywhere (icon chips, nav indicator, panel tabs, CTAs)
Action grid: New Capture / My Spaces / Upload Files / Review & Share
Nav: Home | Spaces | Upload | Review | Account
